# Comprehensive Compiler Construction Project

This extensive documentation details the **Language Processor (Compiler)** developed from scratch as part of the specialized **"Programming Language Translation" (Prevođenje programskih jezika)** course at the **Faculty of Electrical Engineering and Computing (FER)**, University of Zagreb.

This project goes beyond merely implementing a compiler; it involves building the **compiler-generators** themselves. Instead of using off-the-shelf tools like `flex` or `yacc`, this project implements custom generators (`GLA` and `GSA`) that read high-level language specifications (Regular Expressions and Context-Free Grammars) and produce the necessary tables and automata for lexical and syntax analysis.

## 1. Project Overview & Architecture

The compiler targets a subset of the **C programming language** and translates it into **ARM Assembly** code. It operates as a strict pipeline of transformations, reading from `stdin` and writing to `stdout`.

### The Compilation Pipeline

The architecture is divided into four distinct phases, which mimic the structure of industrial-grade compilers (like GCC or Clang):

1. **Lexical Analysis (LA)**:
   - **Generator (`GLA`)**: Analogous to `lex`/`flex`. Reads regex definitions and generates a Finite Automaton (FA).
   - **Runtime (`LA`)**: Uses the generated FA to tokenize the source stream.
2. **Syntax Analysis (SA)**:
   - **Generator (`GSA`)**: Analogous to `yacc`/`bison`. Reads a grammar (CFG) and generates LR parsing tables.
   - **Runtime (`Analizator`)**: Uses the tables to build a Syntax Tree.
3. **Semantic Analysis (`SemA`)**:
   - Performs type checking, name resolution, and scope analysis on the Syntax Tree.
4. **Code Generation (`GK`)**:
   - Translates the verified intermediate representation into ARM Assembly instructions.

## 2. Phase I: Lexical Analysis

### Theoretical Background

Lexical analysis is the process of converting a sequence of characters into a sequence of tokens. A token is a string with an assigned and thus identified meaning.

- **Regular Expressions (Regex):** The patterns for tokens (identifiers, keywords, operators) are defined using formal Regular Expressions.
- **Finite Automata (FA):** Regexes are converted into Non-deterministic Finite Automata (NFA) using algorithms like Thompson's Construction. These are often further converted to Deterministic Finite Automata (DFA) for efficiency.
- **Ambiguity Resolution:**
   - **Longest Match:** If multiple rules match (e.g., `return` vs `ret`), the one consuming the most characters wins.
   - **Priority:** If matches are equal length (e.g., keyword `int` vs identifier `int`), the rule defined first in the specification wins.

### The Generator: GLA (Generator Leksičkog Analizatora)

_Analogous to `lex` or `flex`._

`GLA` is a standalone tool responsible for building the lexer.

- **Input:** It takes a specification file containing regular definitions (e.g., `{digit}`, `{letter}`) and token rules.
- **Process:**
   1. Parses the regular expressions.
   2. Constructs an NFA for each token definition.
   3. Combines these NFAs into a single large automaton that can recognize any valid token.
   4. Serializes this automaton structure (states, transitions, accepting states) into a file (e.g., `automats.txt`).

### The Runtime: LA (Leksički Analizator)

`LA` is the runtime component that performs the actual scanning during compilation.

- **Input:** Source code (`.c` file).
- **Process:** Loads the automaton generated by `GLA`. It simulates the movement through the automaton states as it reads characters from `stdin`.
- **Output:** A stream of tokens (e.g., `IDN 3 main`, `L_PAREN 3 (`) printed to `stdout`.

## 3. Phase II: Syntax Analysis

### Theoretical Background

Syntax analysis verifies that the sequence of tokens conforms to the grammatical structure of the language.

- **Context-Free Grammar (CFG):** The language syntax is defined using productions in Backus-Naur Form (BNF).
   - _Example:_ `<statement> ::= IF LPAREN <expression> RPAREN <statement>`
- **LR Parsing (Bottom-Up):** The project implements a bottom-up parsing strategy, likely **SLR(1)** or **LALR(1)** (Look-Ahead LR).
   - **Shift:** Pushing tokens onto a stack.
   - **Reduce:** Recognizing a sequence on the stack matches a RHS of a production and replacing it with the LHS non-terminal.
- **Parsing Tables:** The efficiency of LR parsing comes from pre-computed tables:
   - **Action Table:** Tells the parser whether to Shift, Reduce, Accept, or Error based on the current state and lookahead token.
   - **Goto Table:** Determines the next state after a reduction.

### The Generator: GSA (Generator Sintaksnog Analizatora)

_Analogous to `yacc` or `bison`._

`GSA` is the most complex component of the front-end. It automates the creation of the parser.

- **Input:** A grammar definition file specifying terminals, non-terminals, and production rules.
- **Process:**
   1. Computes **First and Follow sets** for all non-terminals to determine lookaheads.
   2. Constructs the **Canonical Collection of LR(0) or LR(1) items**. Each "item" represents a possible state of parsing a specific production rule (e.g., `A -> α.β`).
   3. Builds the **Action and Goto tables** by analyzing transitions between item sets.
   4. Detects **Shift-Reduce** or **Reduce-Reduce** conflicts (ambiguities in the grammar).
   5. Serializes the resulting tables into a file (e.g., `kan.san` or `sim.san`).

### The Runtime: SA (Sintaksni Analizator)

The `Analizator` is the pushdown automaton (PDA) driver.

- **Input:** Token stream from `LA`.
- **Process:**
   1. Reads tokens and queries the Action table provided by `GSA`.
   2. Performs Shift and Reduce operations.
   3. Builds the **Generative Syntax Tree (GST)** as it reduces.
   4. Performs error recovery if the syntax is invalid (e.g., panic mode or discarding tokens).
- **Output:** A textual representation of the Syntax Tree, indented to show depth.

## 4. Phase III: Semantic Analysis (SemA)

### Theoretical Background

Simply parsing code is not enough; `x + y` is valid syntax, but invalid semantics if `x` is an integer and `y` is a function. SemA ensures type safety and scope correctness.

- **Scope Resolution:** Validates that identifiers are declared before use. It handles complex scoping rules, such as local variables shadowing global ones.
- **Type Checking:** Enforces the type system (Start/End checks, array indexing validity, function argument matching).
- **L-value Validity:** Ensures that assignments are only made to assignable memory locations (variables, array elements), not constants or expressions.

### Implementation

The `SemA` module traverses the syntax tree produced by the `SA`.

- **Symbol Tables:** It constructs a hierarchy of symbol tables (one for global, others for local scopes).
- **AST Decoration:** It annotates the tree nodes with type information (e.g., marking a node as `type: INT`, `l-value: true`).
- **Function Linking:** It validates that all declared functions are eventually defined (implemented) and matches signatures across translation units (simulated).

## 5. Phase IV: Code Generation (GK)

### Theoretical Background

The final phase translates the validated, high-level AST into low-level machine code for the **ARM architecture**.

- **Stack Frames (Activation Records):**
   - Manages the runtime stack for function calls.
   - Saves/restores the **Link Register (LR)** and **Frame Pointer (FP)**.
   - Allocates space for local variables and parameters.
- **Parameter Passing:** Implements the calling convention (passing arguments in `R0`-`R3` and stack).
- **Control Flow:** Translates `if`, `while`, `for`, and `return` into ARM branching instructions (`B`, `BL`, `CMP`, `BEQ`, `ADD`, `SUB`, `LDR`, `STR`).

### Implementation

The `GK` module emits the assembly code.

- **Register Usage:** Utilizes `R0` as the accumulator/result register, `R13` as Stack Pointer (SP), and `R14` as Link Register (LR).
- **Code Template:**
   - **Prolog:** `PUSH {R0-R3, LR}` ensuring context preservation.
   - **Body:** Generating instructions for expressions and logic.
   - **Epilog:** `POP {R0-R3, PC}` (or similar) to return from functions.
- **Simulator:** The generated `.s` files are tested using the **Unicorn Engine** via `simulator.ipynb`, which emulates the ARM CPU to verify the correctness of the register states after execution.

## How to Run the Entire Pipeline

The true power of this project is chaining these custom tools together.

```bash
# 1. Generate the Lexical Analyzer definition
java LA.GLA < lang_spec.lan > automats.txt

# 2. Generate the Syntax Analyzer tables
java SA.GSA < grammar_spec.san > kan.san

# 3. Run the pipeline on source code
java LA.LA < source.c | java SA.Analizator | java SemA.SemA | java GK.GeneratorKoda > a.s
```

## Simulation & Testing

To verify the generated assembly:

1. Ensure you have the ARM GNU Toolchain installed.
2. Open `GK/simulator.ipynb` in VS Code or Jupyter.
3. The notebook will:
   - Assemble the code: `arm-none-eabi-as`
   - Link the object file: `arm-none-eabi-ld`
   - Run the binary in the customized Unicorn emulator.
   - Check if register `R6` holds the expected return value from the test case.
